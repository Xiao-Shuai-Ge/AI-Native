# AI Native 多智能体协作平台项目报告

## 1. 项目目标

本项目面向两周课程作业，目标是构建一个可本地运行、可恢复、可演示的 AI Native 多智能体协作平台。用户提交研究类任务后，平台可以使用 LangGraph、CrewAI 或自动路由完成“资料收集 -> 内容分析 -> 报告生成”，并在 Web 页面中展示状态、步骤、工具调用、Token 指标和最终报告。

项目优先级为：

1. 可运行、可恢复、可重复演示。
2. 覆盖 P0 验收项。
3. 代码清晰、测试可靠、文档完整。
4. 界面美观和扩展功能。

## 2. 技术架构

核心技术栈：

- 后端：Python 3.12、FastAPI、Pydantic v2。
- Agent 编排：LangGraph 与 CrewAI。
- 耐久执行：Dapr Workflow、Dapr State、Dapr Pub/Sub、Dapr Service Invocation。
- 模型接入：统一 `LLMClient`，适配 DeepSeek、Ollama、OpenAI、Claude。
- 工具协议：MCP Python SDK，独立 MCP Server。
- 存储：Redis 保存 Dapr 状态和短期会话，PostgreSQL 保存业务历史、步骤、审计和偏好。
- 前端：React、TypeScript、Vite、Tailwind CSS。
- 可观测性：OpenTelemetry、Jaeger、Prometheus。
- 部署：Docker Compose。

## 3. 核心模块

### 3.1 任务 API 与 Workflow

FastAPI 只负责 HTTP、SSE 与请求响应模型。创建任务后 API 写入 PostgreSQL 与 Dapr State，然后调度 Dapr Workflow。Workflow 只做确定性编排，所有数据库、LLM、MCP、网络和时间相关操作都放在 Activity 中。

任务状态机：

```text
queued -> running
running -> paused | succeeded | failed | cancelled
paused -> running | cancelled
failed -> running
```

### 3.2 LangGraph 编排

LangGraph 使用固定 `StateGraph`：

```text
START -> plan -> select_roles -> researcher? -> analyst? -> writer -> persist_result -> END
```

特点：

- 共享 `TaskState`。
- 条件边根据 planner 选择的角色跳过部分节点。
- `writer` 固定保留，保证生成最终报告。
- `thread_id=task_id`，checkpoint 通过 `DaprCheckpointSaver` 写入 Dapr State。

### 3.3 CrewAI 编排

CrewAI 使用共享角色注册表构建 researcher、analyst、writer 的 Agent/Task/Crew。为了符合 Dapr Workflow 的重试粒度，每个角色步骤封装成独立 Activity，避免失败时重跑整个 Crew。

两个引擎共用：

- 角色配置。
- LLM adapter。
- MCP client。
- 数据库 repository。
- 事件 schema。
- Web UI 与 API 输出模型。

### 3.4 MCP 工具

MCP Server 提供 4 个工具：

| 工具 | 功能 | 安全策略 |
| --- | --- | --- |
| calculator | 白名单数学表达式 | AST 白名单，不直接 eval |
| web_search | 搜索标题、摘要和 URL | 超时、条数和响应大小限制 |
| code_runner | 运行短 Python/Shell 代码 | 独立 Docker sandbox，无网络、非 root、只读根、资源限制 |
| readonly_sql | 演示数据库只读查询 | 单条 SELECT、只读用户、行数和超时限制 |

API/Worker 优先通过 Dapr Service Invocation 调用 MCP app，并透传 trace headers。

### 3.5 Web 控制台

Web 控制台包含三个页面：

1. 新建任务：输入主题，选择自动/LangGraph/CrewAI 和模型参数。
2. 任务详情：展示状态、实际引擎、选择原因、时间线、工具调用、Token、报告、Jaeger/Prometheus 链接。
3. 历史与设置：展示历史任务，编辑三个固定角色与模型参数。

SSE 断线后有限重连，失败后降级为详情轮询。

## 4. 可观测性

结构化日志、审计事件、OpenTelemetry span 和 Prometheus 指标共同覆盖任务执行链路。

稳定 span 名称包括：

- `task.create`
- `workflow.activity.*`
- `router.select_engine`
- `langgraph.node.*`
- `crewai.task.*`
- `llm.chat`
- `mcp.tool.call`

Prometheus 指标按低基数标签 `engine`、`provider` 汇总任务完成数、耗时、Token 和工具调用结果。

## 5. 测试与验收

自动化测试覆盖：

- Agent schema 与节点输出。
- LangGraph 图构建、条件边和 checkpoint。
- CrewAI Agent/Task/Crew 构建与结果转换。
- 自动路由、fallback 和手动覆盖。
- 任务状态迁移。
- MCP 4 工具的成功、非法输入和超时。
- API 核心接口。
- Web UI 创建、详情、SSE 降级、设置和指标展示。

真实验收脚本：

```bash
uv run python scripts/day10_acceptance.py --engines langgraph,crewai,auto --concurrency 10 --output docs/day10-acceptance-results.json
```

脚本会检查 `/health`、`/ready`、`/api/providers`、`/api/tools`，并分别创建三种引擎任务，最后执行轻量并发任务。

## 6. 部署与演示

启动完整栈：

```bash
docker compose up -d --build api api-daprd worker worker-daprd mcp-server mcp-server-daprd web redis postgres placement scheduler jaeger prometheus
```

访问入口：

- Web UI: `http://localhost:5173`
- API: `http://localhost:8000`
- Jaeger: `http://localhost:16686`
- Prometheus: `http://localhost:9090`

## 7. 已知限制

- 默认 Compose 演示为单 worker。Dapr sidecar 需要与 app 实例一一绑定，直接 `docker compose up --scale worker=2` 不能自动创建第二个 sidecar；生产形态建议迁移到 Kubernetes 或显式定义第二组 worker/sidecar。
- 真实 LLM smoke test 依赖本地 Ollama 或 DeepSeek key，默认 CI 使用 fake/mock LLM。
- 外部搜索 API 受网络与额度影响，测试中使用 mock 保证稳定。
- 项目不实现拖拽式编排、任意角色创建、多租户、复杂权限和向量数据库长期记忆。

## 8. 总结

项目已经形成从 Web、API、Dapr Workflow、双 Agent 引擎、MCP 工具、存储到可观测性的完整闭环。课程演示重点应放在可运行、可恢复、可观测和双编排对比，而不是扩展为通用商业平台。
